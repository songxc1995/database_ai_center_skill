# Database AI Center

## Overview
Use this skill as the primary alert-analysis workflow for Database AI Center.

This skill performs the final diagnosis. It may use `zabbix-readonly` as a supporting skill when database-side evidence is insufficient or when host-side performance context is required.
It relies on `GET /alerts` and `GET /ai/context/{instance_id}` only. It does not require direct calls to `/metrics/{id}/latest` or `/metrics/{id}/series`.

## Required Config
Read these values from environment variables or provided runtime config:

- `PROJECT_API_BASE_URL`
- `PROJECT_API_KEY`

Optional values:

- `PROJECT_WEBHOOK_TARGET_ID`
- `PROJECT_ALERT_ID`
- `PROJECT_ALERT_LIMIT`

Example:

```env
PROJECT_API_BASE_URL=http://10.101.240.250:8080/api/v2
PROJECT_API_KEY=replace-with-real-api-key
PROJECT_WEBHOOK_TARGET_ID=1
PROJECT_ALERT_ID=
PROJECT_ALERT_LIMIT=20
```

## Runtime Requirements
The external model platform must support outbound HTTP calls.

No extra model provider is required. The current model performs the diagnosis.

## Workflow
1. Call `GET /alerts?page=1&page_size=<n>`.
2. If `PROJECT_ALERT_ID` is provided, choose that alert; otherwise choose the first item in `items`.
3. Read `instance_id` from the chosen alert.
4. Call `GET /ai/context/{instance_id}`.
5. Produce an initial diagnosis from project API data.
6. If host-side evidence is needed, use `zabbix-readonly`.
7. Merge project evidence and Zabbix evidence.
8. Return the final Chinese structured analysis.
9. If the user asks to dispatch and `PROJECT_WEBHOOK_TARGET_ID` is configured, call `POST /webhooks/{target_id}/dispatch-analysis`.

## When To Use Zabbix Readonly
Use `zabbix-readonly` only when at least one of these is true:

- The alert may be related to CPU, memory, disk, I/O, or OS-side resource pressure.
- The current database-side evidence is insufficient to support a confident diagnosis.
- The user explicitly asks for server-side performance context.

Do not use `zabbix-readonly` when:

- The current database-side evidence is already sufficient.
- No host-side lookup information is available.
- The alert is clearly explained by database-internal evidence alone.

## Zabbix Target Resolution
When using `zabbix-readonly`, pass these fields in this order:

- `host_ip`
- optional `host_name` for disambiguation
- optional `zabbix_hostid` if already known

If `host_ip` matches multiple hosts, do not guess.
If `host_name` is available, use it to narrow the candidate set.
If ambiguity remains, continue the database-side analysis and explicitly note that Zabbix host resolution was ambiguous.

## Analysis Rules
- Use only returned API data and returned Zabbix data.
- Do not invent SQL, logs, topology events, or metrics not present in the payloads.
- Keep the analysis in Chinese.
- `severity` must be one of `low|medium|high|critical`.
- Prefer database-specific reasoning:
  - TiDB: PD, TiKV, Region/Leader distribution, hotspot tables, region imbalance, store health
  - MySQL: connections, active threads, slow SQL pressure, locks, replication lag, storage pressure
  - PostgreSQL: sessions, locks, long transactions, replication lag, replication slot lag, storage pressure
  - Oracle: sessions, long-running sessions, wait events, tablespace, archive/Data Guard lag, I/O pressure
  - ClickHouse: replication queue, replication lag, readonly replicas, merge/fetch pressure, storage pressure
- When project API responses already include `diagnosis_report`, prefer reading `diagnosis_report.summary`, `diagnosis_report.severity`, `diagnosis_report.root_cause`, `diagnosis_report.evidence`, and `diagnosis_report.recommendations`.
- When `root_cause_candidates` or `explainability` is present in AI context, treat them as the highest-value structured evidence and prioritize them over free-form guessing.
- If `latest_metrics` explicitly contains `replication_lag_seconds`, storage pressure signals, or Oracle `oracle_long_running_sessions`, reflect those facts directly in the diagnosis.
- For Oracle, do not interpret `slow_queries` as a slow-log style metric. On older deployments it may be a compatibility alias for long-running active sessions.
- If `diagnosis_report` is absent, fall back to legacy fields such as `risk_level` and `actions`.
- Tolerate mixed-version deployments: newer deployments may expose richer AI context fields, while older deployments may only expose coarse summary fields.
- Zabbix is supporting evidence only. Final `summary`, `root_cause`, and `recommendations` must still be generated by this skill.

## API Contract
Auth header:

```text
X-API-Key: <PROJECT_API_KEY>
```

Base URL example:

```text
http://10.101.240.250:8080/api/v2
```

### 1. List Alerts

```http
GET /alerts?page=1&page_size=20
```

Expected shape:

```json
{
  "items": [
    {
      "id": 1,
      "instance_id": 2,
      "rule_id": "tikv_region_imbalance_high",
      "severity": "critical",
      "status": "active",
      "message": "TiKV region distribution is imbalanced: 76.54",
      "evidence": {
        "metric": "tikv_region_imbalance_pct",
        "threshold": 70.0,
        "actual": 76.54
      }
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 20,
  "has_next": false
}
```

### 2. Get AI Context

```http
GET /ai/context/{instance_id}
```

Useful fields:

- `instance_profile.type`
- `instance_profile.name`
- `health_summary.total_score`
- `health_summary.grade`
- `active_alerts`
- `root_cause_candidates`
- `explainability`
- `latest_metrics`

### 3. Dispatch Analysis

```http
POST /webhooks/{target_id}/dispatch-analysis
```

## Output Contract
Return this structure:

```json
{
  "alert": {
    "id": 1,
    "instance_id": 2,
    "rule_id": "tikv_region_imbalance_high",
    "severity": "critical",
    "status": "active",
    "message": "TiKV region distribution is imbalanced: 76.54"
  },
  "analysis": {
    "summary": "......",
    "severity": "critical",
    "root_cause": "......",
    "evidence": [
      "Database AI Center: ......",
      "Zabbix: ......"
    ],
    "recommendations": [
      {"action": "......"},
      {"action": "......"}
    ]
  },
  "dispatch": null
}
```

Rules:

- `dispatch` is `null` when no webhook call is made.
- `summary` must mention the alert message.
- Include threshold and actual value when available.
- `evidence` should be short factual Chinese strings derived from API payloads.
- When Zabbix data is used, prefix the evidence with `Zabbix:`.
- `recommendations` should contain 3 to 4 short Chinese action items as objects with at least `action`.
- When calling `POST /webhooks/{target_id}/dispatch-analysis`, send the normalized `analysis` shape with `summary`, `severity`, `root_cause`, `evidence`, and `recommendations`.

## Failure Handling
- If `/alerts` returns `items: []`, stop and report that there is no live alert to analyze.
- If the chosen alert has no valid `instance_id`, stop and report malformed alert payload.
- If `/ai/context/{instance_id}` fails, return the alert summary and explicitly note context retrieval failure.
- If Zabbix lookup fails, keep the project API analysis and explicitly note that host-side performance context was unavailable.
- If dispatch fails, keep the analysis output and report dispatch failure separately.
